RESEARCH TREE UPDATE AND CLEANUP PLAN
=====================================

Purpose
-------
This file is the working reference for the research-tree pipeline.
It documents:

1. The current flow used to build a long-form answer
2. The main code locations involved
3. What has already been fixed
4. What still needs cleanup
5. The recommended order for future changes

This should be updated whenever the research-tree pipeline changes.


CURRENT ROLE OF THE RESEARCH TREE
---------------------------------
The research tree is still used for long-form answer generation.

It is NOT used for the normal PDF/project/all search box in the frontend.

There are now two separate product paths:

1. Search
   - Fast retrieval over Elasticsearch
   - Scoped to one PDF, one project, or all documents
   - Uses backend/app/routers/query.py

2. Build Answer
   - Runs the research-tree pipeline
   - Builds a structured long-form answer
   - Uses backend/app/routers/agent.py

CURRENT RESEARCH TREE FLOW
--------------------------
The current long-form pipeline inside agent.py follows this shape:

1. Resolve scope
   - Scope can be:
     a) all documents
     b) one project
     c) one PDF
   - Resolved through backend/app/utils/document_scope.py

2. Retrieve initial chunks
   - Initial vector retrieval is done through:
     backend/app/utils/agent/search_chunks.py
   - This now respects the selected scope

3. Create root tree
   - A root ResearchTree is created
   - Root title starts from the user query
   - Scope is stored on the tree

4. Persist root tree and attach initial chunks
   - Tree is saved through ResearchTreeRepository
   - Initial chunks are stored and attached to the root node

5. Generate root subquestions
   - Subquestions are generated from root chunks
   - Uses backend/app/utils/agent/subquestions.py

6. Attach root subquestions to the tree
   - These are routed to the best node currently available
   - At this stage, that is usually still the root or top-level candidates

7. Generate outline from the tree
   - Uses backend/app/utils/agent/outline.py
   - The outline now sees the generated subquestions because they are attached before outline generation

8. Apply outline back into the tree
   - Outline sections are mapped into ResearchNode children
   - Outline questions are attached to those nodes

9. Process top-level sections recursively
   - Each section may:
     a) enrich with more chunks
     b) generate new expansion subquestions
     c) deepen into subnodes if justified
     d) write the section text
   - Uses backend/app/utils/agent/expander.py

10. Generate executive summary and overall conclusion
    - Uses backend/app/utils/agent/writer.py

11. Finalize article text from the tree
    - Uses backend/app/utils/agent/finalizer.py

12. Return structured result
    - title
    - abstract
    - outline
    - sections
    - article


WHAT HAS BEEN FIXED ALREADY
---------------------------
1. Scope is now part of the tree flow
   - The research tree can now be constrained to:
     a) one PDF
     b) one project
     c) all documents
   - This applies to initial retrieval and later expansion retrieval

2. Build Answer no longer relies on one fragile long HTTP request
   - The frontend now starts an answer run and polls for status
   - This avoids connection reset / failed to fetch issues in the browser

3. Root subquestions are now attached before outline generation
   - This is the first real research-tree correctness fix
   - Before this change, subquestions were generated but only attached after the outline step
   - That meant the outline was missing one of its main inputs

4. ResearchTreeRepository.save() no longer commits internally
   - Transaction ownership now stays with the orchestration layer
   - This removes a hidden side effect from tree persistence

5. full_run now keeps the working tree in memory longer
   - Initial chunks, root subquestions, outline generation, outline application,
     section collection, and final article synthesis no longer require repeated
     full tree reloads between each step
   - Database writes still happen where helper layers require them, but the
     main orchestration is less round-trip heavy

6. The frontend can now follow an answer run live
   - Build Answer now exposes backend stage updates while the run is active
   - The frontend polls the persisted tree and renders a live nested tree view
     during construction

7. The tree now uses an adaptive research plan
   - Root retrieval, root subquestion count, outline size, section retrieval,
     section context size, section length guidance, and desired depth are now
     derived from query complexity and scope breadth
   - The pipeline is no longer supposed to use one near-constant budget for
     every question

8. Node refinement is now partially evidence-aware
   - Files:
     backend/app/utils/agent/controller.py
     backend/app/utils/agent/expander.py
   - Nodes now build a local evidence profile from retrieved chunks
   - That profile now influences:
     a) section context size
     b) per-node subquestion target
     c) section length hint
     d) whether a node should even attempt to deepen
   - This is the first implementation slice of the Research Tree V2 direction

9. Root planning is now refined from initial retrieved evidence
   - Files:
     backend/app/utils/agent/planning.py
     backend/app/routers/agent.py
   - The first retrieval now adjusts key budgets before subquestions and outline:
     a) root_subquestion_target
     b) outline_target_sections
     c) section_top_k and section_subquestion_target
     d) section context and summary budget
   - This reduces the tendency to produce the same fixed shape for unrelated queries

10. Structural deepening is now automatic in recursive processing
    - File:
      backend/app/utils/agent/expander.py
    - When refinement approves deepening, novel questions are clustered into
      child subnodes and attached automatically during full_run
    - This moves deepening behavior from a mostly manual route toward the
      intended local refinement loop

11. Plan budgets are now less rigid and more candidate-vs-context aware
    - File:
      backend/app/utils/agent/planning.py
    - root_top_k now behaves as a broader candidate-retrieval budget
    - root_context_chunks now behaves as a filtered evidence budget for LLM steps
    - simple single-document queries can now legitimately use smaller question/section plans

12. Node enrichment now performs a second retrieval pass from generated expansion questions
    - File:
      backend/app/utils/agent/expander.py
    - Each node now follows a closer local loop:
      retrieve -> ask follow-up questions -> retrieve again -> write/deepen decision
    - This reduces reliance on one single retrieval pass per node

13. Node-level relevance gate now checks alignment with the original question
    - Files:
      backend/app/utils/agent/writer.py
      backend/app/utils/agent/expander.py
    - Before writing/deepening a node, the pipeline now runs an explicit
      alignment check versus the root question
    - Off-topic branches are skipped instead of being expanded into long text

14. Output style is now configurable end-to-end
    - Files:
      backend/app/static/index.html
      backend/app/static/app.js
      backend/app/routers/agent.py
      backend/app/utils/agent/outline.py
      backend/app/utils/agent/writer.py
      backend/app/models/research_tree.py
    - Build Answer now supports style selection:
      a) scientific article
      b) blogpost
      c) newspaper
    - The selected style now flows into planning metadata and writing prompts

15. Sibling/subsection overlap consolidation now runs before final synthesis
    - Files:
      backend/app/utils/agent/overlap.py
      backend/app/routers/agent.py
    - The pipeline now compares sibling and sub-sibling section content for overlap
    - Decision policy currently supports:
      a) rewrite secondary section to keep only unique angles
      b) prune secondary section when no unique value remains
    - Consolidation decisions are logged and included in run result payload


CURRENT WEAKNESSES
------------------
1. The top-level planner is still heuristic-based
   - File: backend/app/utils/agent/planning.py
   - The tree now adapts to query/scope breadth, and nodes now react to local
     evidence density, but the run-level planner still does not inspect
     retrieval scores or source diversity deeply enough

2. Structural cleanup is still partial (prune/rewrite exists; merge/split are not complete)
   - Files:
     backend/app/utils/agent/controller.py
     backend/app/utils/agent/expander.py
     backend/app/utils/agent/overlap.py
   - We now have post-write overlap pruning/rewriting, but still do not have
     full merge/split restructuring at planning/refinement time

3. Helper layers still open their own DB sessions during recursive processing
   - Files:
      backend/app/utils/agent/expander.py
      backend/app/utils/agent/controller.py
      backend/app/utils/agent/writer.py
   - full_run is leaner now, but the recursive pipeline still hides a lot of
     DB traffic behind helper calls

4. The pipeline mixes orchestration, persistence, and LLM logic in one large router file
   - File: backend/app/routers/agent.py
   - This should eventually become a service layer

5. Root abstract handling is weak
   - The returned abstract often comes from root_node.content
   - But root content is not clearly produced as a true abstract stage

6. Summary and conclusion generation are too similar
   - File: backend/app/utils/agent/writer.py
   - write_summary() and write_conclusion() are almost the same in structure
   - They should become clearly separate writing goals

7. Initial root chunk provenance is only partially normalized
   - Root retrieval in full_run now keeps available page/source metadata
   - Some fallback paths still use text-only chunk records
   - Provenance handling should be made consistent across all entry routes

8. Tree persistence is append/update oriented, not fully pruning old branches
   - File: backend/app/repositories/research_tree_repo.py
   - This can leave stale nodes if a structure changes significantly

9. full_run still reopens sessions around major phases
   - File: backend/app/routers/agent.py
   - The worst repeated tree reloads are gone, but there are still separate
     sessions for the setup, outline persistence, and root finalization phases


RECOMMENDED CLEANUP ORDER
-------------------------
Phase 1: Correctness
1. Done: attach root subquestions before outline generation
2. Generate a real abstract stage instead of using root content ambiguously
3. Separate summary and conclusion prompts cleanly

Phase 2: Transaction cleanup
1. Done: remove internal commit from ResearchTreeRepository.save()
2. Keep the orchestration layer as the owner of commit boundaries
3. Make tree operations more predictable in one run

Phase 3: Session and reload reduction
1. Done: reduce unnecessary tree reloads in full_run
2. Keep pushing DB session ownership out of helper layers
3. Persist checkpoints only where needed

Phase 4: Retrieval quality and provenance
1. Done: adaptive retrieval and writing budgets now vary with query/scope breadth
2. Done: keep available page/source metadata on initial root chunks in full_run
3. Improve retrieval payloads used for writing
4. Tune using source diversity / retrieval scores, not only heuristics

Phase 5: Code structure
1. Move large orchestration logic out of backend/app/routers/agent.py
2. Introduce a dedicated agent service layer
3. Keep routers thin and operational

Phase 6: Optional product improvements
1. Add multiple answer modes:
   - summary
   - article
   - article depth levels
2. Add answer-run history in the frontend
3. Add export options directly from Build Answer results


RESEARCH TREE V2 DIRECTION
--------------------------
The agreed next architecture is not a global restart loop.

The tree should evolve toward a bounded local refinement loop per node or subtree.

The intended shape is:

1. Scope
   - Decide what corpus is allowed
   - One PDF, one project, or all documents

2. Run policy
   - Decide overall ambition, budget, and max depth
   - Keep this smaller than the current flat ResearchPlan
   - This is strategy, not per-node execution

3. Seed tree
   - Query -> root chunks -> root questions -> draft outline
   - This creates the first usable structure

4. Local refinement loop
   - For each node:
     a) retrieve chunks
     b) generate follow-up questions
     c) retrieve again if justified
     d) score evidence quality
     e) prune weak chunks and irrelevant questions
     f) decide stop / deepen / merge / split
   - This loop should be local to the node or subtree
   - It should not restart the whole tree

5. Global consolidation
   - Remove overlap between sibling sections
   - Merge thin or duplicate sections
   - Rebalance the outline once if needed
   - Avoid repeated whole-tree rebuilds

6. Writing
   - Write sections only after the node or subtree is stable enough
   - Then synthesize executive summary and overall conclusion


WHY THIS DIRECTION
------------------
This should make the system:

1. More intelligent
   - Different sibling nodes can behave differently based on their own evidence

2. More stable
   - No expensive and noisy global restart cycle after every local change

3. Easier to reason about
   - Top level owns strategy
   - Lower levels own execution decisions

4. Easier to control
   - The loop can stop when a node is good enough instead of always expanding


NON-GOAL
--------
Do not build a blind global loop like:

query -> chunks -> questions -> chunks -> outline cleanup -> restart everything

That would likely cause:
- cost blowups
- unstable outlines
- repeated churn in the persisted tree
- harder debugging


TARGET CONTROL RULES
--------------------
The future controller should use explicit stop conditions.

Examples:

1. No sufficiently novel questions
2. No materially new chunks from another retrieval pass
3. Evidence overlap too high with sibling nodes
4. Coverage is already good enough for the node
5. Budget exhausted for that node
6. Max node iterations reached

This should be implemented as a bounded refinement controller, not an open-ended loop.


PARAMETER OWNERSHIP DIRECTION
-----------------------------
Some current flat ResearchPlan parameters should move lower in the system.

Keep at run-policy level:
- scope
- overall budget level
- desired depth
- evidence profile
- query complexity

Keep at phase level:
- root_top_k
- root_context_chunks
- root_subquestion_target
- outline_target_sections
- outline_max_subsections
- summary_context_sections

Move to node-level execution logic:
- section_top_k
- section_context_chunks
- section_subquestion_target
- section_length_hint
- min_novel_questions_to_deepen

Reason:
- these are better decided per node from actual evidence, not once globally


V2 CONTROLLER IDEA
------------------
The controller should become the main owner of local refinement decisions.

High-level responsibilities:

1. Evaluate node evidence quality
2. Decide whether to deepen
3. Detect overlap with sibling sections
4. Detect redundant or irrelevant questions
5. Decide whether a node should stop, deepen, split, or merge

This suggests evolving backend/app/utils/agent/controller.py into a real refinement controller,
while keeping backend/app/utils/agent/expander.py focused on executing one node iteration.


NEXT RECOMMENDED CHANGE
-----------------------
The next best execution block is:

Make output style control structure and size, not only wording.

Why this is next:
- current output_style mostly affects writing prompts
- section/depth planning is still largely query/scope driven
- this causes similar tree shapes across styles for the same question

Expected effect:
- blogpost vs scientific vs newspaper will produce visibly different tree geometry
- output length will become predictable by style/mode
- less unnecessary depth for lightweight formats


CURRENT STATE SNAPSHOT (2026-03-06)
-----------------------------------
Confirmed status:
1. Trees are persisted and reloadable by session_id
2. Style selection exists end-to-end (UI -> API -> plan metadata -> prompts)
3. Node-level alignment gate exists and skips off-topic branches

Confirmed gap:
1. Style does not yet strongly steer outline size, depth, and expansion policy
2. Word count is not yet enforced as a hard budget
3. Local refinement still lacks explicit prune/merge/split consolidation rules


NEXT EXECUTION PLAN (ACTIVE)
----------------------------
Step 1: Introduce style/mode geometry policy
- Add explicit output mode:
  a) brief
  b) standard
  c) deep
- Keep output_style for tone:
  a) scientific_article
  b) blogpost
  c) newspaper
- Apply style+mode multipliers to:
  a) outline_target_sections
  b) desired_depth
  c) section_subquestion_target
  d) section_context_chunks

Step 2: Add hard length budgets
- Add run-level and section-level word targets/ranges to ResearchPlan
- Enforce in writing prompts and post-write budget checks
- Add compression pass when section/run exceeds budget

Step 3: Add explicit structural decisions in controller
- Implement local decisions:
  a) prune weak/off-topic node
  b) merge near-duplicate sibling
  c) split overloaded node
  d) deepen only when novelty and evidence justify it

Step 4: Consolidation pass after local refinement
- Remove sibling overlap
- Merge thin duplicate sections
- Ensure final tree is compact for the selected style/mode


ACCEPTANCE CRITERIA FOR THE NEXT STEP
-------------------------------------
When the session/reload cleanup is done:

1. the controller should react not only to query text but also to retrieved evidence shape
2. the controller should be able to prune, merge, or skip weak branches locally
3. local node refinement should not require whole-tree restart cycles
4. The logic should remain transactionally correct
5. full_run should still work end-to-end
6. start_answer_run / polling flow should still work
7. Tree exports should still load the correct persisted state


NOTES FOR FUTURE CHANGES
------------------------
When changing the research-tree pipeline, always check:

1. Does the selected scope still flow through all retrieval steps?
2. Are subquestions attached before outline generation?
3. Are transactions owned by the orchestration layer, not hidden in helpers?
4. Are retrieved chunks keeping enough provenance for debugging and citations?
5. Does Build Answer still work through start + poll instead of one long request?
